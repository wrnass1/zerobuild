# Платформа для pet-проектов без кода

# 1. Описание проекта

## 1.1 Назначение системы

Платформа предназначена для автоматизации запуска pet-проектов без необходимости ручного поиска команды и организации рабочего процесса.

Система объединяет разработчиков, предоставляет каталог идей, выполняет подбор участников по стеку технологий и автоматически создает рабочее пространство с Kanban-доской.

---

## 1.2 Целевая аудитория

- Студенты IT-направлений  
- Junior / Middle разработчики  
- Участники хакатонов  
- Специалисты, развивающие портфолио  

---

## 1.3 Проблематика

- Сложно найти команду для pet-проекта  
- Нет централизованной базы идей  
- Организационный хаос на старте  
- Отсутствует единая среда для совместной работы  

---

# 2. Архитектура

Микросервисная архитектура на FastAPI.

## Сервисы:

- **Auth Service** — управление пользователями и авторизация  
- **Idea Service** — управление идеями проектов  
- **Matching Service** — подбор участников по стеку  
- **Kanban Service** — управление досками и задачами  

---

## Базы данных

- PostgreSQL — основное хранилище данных  
- Redis — кэширование и очереди  

Рекомендуемый подход: отдельная база данных для каждого сервиса.

---

# 3. Функциональные требования

## 3.1 Auth Service

Система должна обеспечивать:

- Регистрацию пользователя  
- Авторизацию (JWT)  
- Хранение профиля  
- Указание стека технологий  
- Редактирование профиля  

### Профиль пользователя включает:

- Имя  
- Email  
- Стек технологий  
- Уровень (Junior / Middle)  
- Описание  
- Список проектов  

---

## 3.2 Idea Service

Система должна обеспечивать:

- Создание идеи  
- Просмотр каталога  
- Фильтрацию по стеку  
- Фильтрацию по сложности  
- Поиск по названию  
- Редактирование и удаление идеи  

### Структура идеи:

- Название  
- Описание  
- Предполагаемый стек  
- Уровень сложности  
- Количество участников  
- Статус (открыта / в работе / завершена)  

---

## 3.3 Matching Service

Назначение: автоматический подбор участников по совпадению стека.

### Требования:

- Анализ пересечения стеков  
- Расчет коэффициента совпадения  
- Сортировка по степени релевантности  
- Отправка приглашения в проект  

### MVP-алгоритм:

Процент совпадения технологий между участником и требуемым стеком проекта.

---

## 3.4 Kanban Service

Система должна обеспечивать:

- Автоматическое создание доски при создании проекта  
- Создание колонок (To Do / In Progress / Done)  
- Создание задач  
- Перемещение задач между колонками  
- Назначение исполнителя  
- Изменение статуса  

### Структура задачи:

- Название  
- Описание  
- Исполнитель  
- Статус  
- Дедлайн (опционально)  

---

# 4. API (примерная структура)

## Auth Service

POST /register
POST /login
GET /profile
PUT /profile

## Idea Service
GET /ideas
POST /ideas
GET /ideas/{id}
PUT /ideas/{id}
DELETE /ideas/{id}

## Matching Service
GET /match/{idea_id}
POST /invite

## Kanban Service
POST /boards
GET /boards/{id}
POST /tasks
PUT /tasks/{id}


---

# 5. Пользовательские сценарии

### Сценарий 1
Пользователь регистрируется → указывает стек → получает список релевантных идей.

### Сценарий 2
Создает идею → система предлагает подходящих участников → формируется команда.

### Сценарий 3
После формирования команды автоматически создается Kanban-доска.

---

# 6. Нефункциональные требования

- Контейнеризация через Docker  
- Запуск через docker-compose  
- Асинхронный FastAPI  
- JWT-авторизация  
- Разграничение прав доступа  
- Индексация БД для ускорения поиска  

---

# 7. Ограничения

- MVP без сложной ML-модели  
- Без интеграции с внешними API  
- Упор на backend-архитектуру  

---

# 8. Definition of Done

- Все сервисы запускаются через docker-compose  
- Регистрация и авторизация работают  
- Идея создается и отображается  
- Matching возвращает корректный список  
- Kanban-доска создается автоматически  
- Демонстрация полного пользовательского сценария
